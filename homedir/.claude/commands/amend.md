# amend

## 目的

直前のコミットメッセージを修正する。
ただし **意図の明確化と履歴の一貫性を重視** し、次の原則に従う。

- 不要な amend による履歴破壊を避ける
- 小さな修正（誤字、コメント修正、軽微な差分追加など）に限定
- 修正理由を明確にし、コミットメッセージにも反映する

## 基本原則
- コミットメッセージは平易な英文にする
- 1つのコミットには1つの論理的な変更のみを含める
- 関連する変更はまとめて、無関係な変更は分離する
- コミットメッセージから変更内容が明確に理解できる粒度にする
- コミットメッセージは変更点の要約ではなく、コミットの意図を書くこと

## 手順

git rebase の操作はユーザに任せ、ここではコミットメッセージの修正のみに留めてください。

1. **現在の状態確認**
   - `git status` と `git log -1 -p` を実行し、直前のコミット内容を確認する
2. 変更ファイル・内容を分類（例：機能追加、バグ修正、リファクタリング、ドキュメント更新 等）
3. 各カテゴリに応じて、短く意図を含んだメッセージ案を生成
4. ユーザーに最終確認を求め、`git commit --amend` を実行

変更の意図が分からない場合はユーザに質問してください。その際、質問はひとつづつ行ってください。

## コミットメッセージテンプレート

```
<type>(<scope>): <subject>

<description>
```

- `<type>` は以下のいずれか:
    - feat: 新機能追加
    - fix: バグ修正
    - refactor: リファクタリング
    - docs: ドキュメント
    - chore: その他（ビルド設定、メンテナンス等）
- `<scope>` は任意
    - multiproject 構成の場合は project(directory) 名 を含める
- <subject> は 50 文字以内で意図を簡潔に表現
- <description> は「なぜこの変更を行ったか」を 1〜3 行で説明
