# commit

## 基本原則
- コミットメッセージは平易な英文にする
- 1つのコミットには1つの論理的な変更のみを含める
- 関連する変更はまとめて、無関係な変更は分離する
- コミットメッセージから変更内容が明確に理解できる粒度にする
- コミットメッセージは変更点の要約ではなく、コミットの意図を書くこと

### コミット粒度 判断基準

1. **機能的な独立性**: 他の変更なしに動作するか
2. **レビューの容易さ**: 差分が理解しやすいサイズか
3. **履歴の追跡性**: 後から特定の変更を見つけやすいか

## 手順

以下の手順で変更を解析し、適切な粒度のコミットを作成します:

1. `git status` による変更一覧
2. `git diff` による差分解析
3. 変更ファイル・内容を分類（例：機能追加、バグ修正、リファクタリング、ドキュメント更新 等）
4. 各カテゴリに応じて、短く意図を含んだメッセージ案を生成
5. ユーザーに最終確認を求め、ステージング → commit を実行
6. 必要に応じて差分ブロックを分割し、`git add -p` 相当の粒度でステージング
7. `git log --oneline -n 20` を参照し、過去の命名規約に沿うようにメッセージを整形

変更の意図が分からない場合はユーザに質問してください。その際、質問はひとつづつ行ってください。

## コミットメッセージテンプレート

```
<type>(<scope>): <subject>

<description>
```

- `<type>` は以下のいずれか:
    - feat: 新機能追加
    - fix: バグ修正
    - refactor: リファクタリング
    - docs: ドキュメント
    - chore: その他（ビルド設定、メンテナンス等）
- `<scope>` は任意
    - multiproject 構成の場合は project(directory) 名 を含める
- <subject> は 50 文字以内で意図を簡潔に表現
- <description> は「なぜこの変更を行ったか」を 1〜3 行で説明
